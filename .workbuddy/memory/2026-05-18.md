## 鉴权改造：URL 强制登录 + 刷新保持登录

### 需求
- 浏览器输入任意 URL（任何形式拼接） → 一律先跳 `#/login`
- 登录态下 F5 刷新当前标签 → 保持登录、原页面不变、不出 bug

### 关键洞察
区分"新开标签/新会话" vs "同标签 F5"：用 `sessionStorage`。它在同标签 F5 时保留，新开标签/新窗口/重启浏览器时清空，正好用来识别。

### 改动（最小）
1. **`frontend/src/stores/user.js`** —— 持久化 userInfo
   - 初始值 `ref(JSON.parse(localStorage.getItem('userInfo') || 'null'))`，try/catch 兜底脏数据
   - `setUserInfo` 在 normalizeEncoding 后写 `localStorage.userInfo`（null 则 remove）
   - `logout` 同时清 token + userInfo
2. **`frontend/src/main.js`** —— 在 createApp 之前插入新会话检测
   - sessionStorage key `app_session_initialized`
   - 不存在（新会话）→ set flag、清 token+userInfo、`location.replace(... + '#/login')`（已在 #/login 时不重复跳避免死循环）
   - 已存在（同标签刷新）→ 整段跳过，让 store 从 localStorage 恢复 + router beforeEach 兜底

### 双层保险
1. main.js 在 app 创建前清 token —— 新会话已无 token
2. router.beforeEach 现有的"无 token + 非 /login → 跳 /login"逻辑兜底

### 关键陷阱（之前的 bug 来源）
- **userInfo 不持久化**：刷新后 token 在但 userInfo 丢失，beforeEach 中 `to.meta.roles` 校验拿不到 role，会乱跳。当前 router 代码用了 `if (userRole && !roles.includes(...))` 短路保护，userRole undefined 时不误拦，但用户体验仍异常（菜单空、信息丢）。持久化 userInfo 后这个问题彻底解决。
- **副作用扫描**：`localStorage` 在 src 里 14 处引用，token 写入只在 `stores/user.js:setToken` 一处（无旁路），`services/request.js:17` 拦截器直读 token —— 新会话清 token 后下次请求自然不带 Bearer，行为一致无残留。

### 验收（QA 静态推演 + dev server 烟雾）
- 4 大场景全 ✅：新开标签 + 无 token → /login；新开标签 + 伪造 token → 清掉跳 /login；学员 F5 留在 /courses；管理员 F5 留在 /admin
- `npm run dev` 启动 OK，无编译错误

### 后续可选加固（非本次必做）
- 后端补 `GET /api/auth/me`，App.vue onMounted 同步 userInfo（防服务端 token 已吊销）
- 多标签 storage 事件同步登出
- `services/request.js` 改走 store 而非直读 localStorage

## 取消预约三连弹窗 / 取消后再预约 Duplicate / 禁用账号仍能登录

### Bug 单
1. 取消预约后弹"取消预约成功 + 该预约状态不允许取消 + 取消失败"三连
2. 取消后再次预约 → 弹 `Duplicate entry '10-2' for key 'bookings.uk_student_course'`
3. 管理员禁用账号后，账号仍能登录（应提示"账号已被禁用"）

### Root Cause
1. **三连弹窗 = 双击叠加 + 双层弹错**。`request.js` 拦截器对非 `00000` 已经弹过 `message.error(msg)`，但 `MyAppointments/handleCancel` catch 又调一次 `message.error('取消失败')`；用户点取消按钮没置 disabled，第一次成功 → 第二次状态已是 cancelled → 99998 → 拦截器弹"该预约状态不允许取消" + catch 弹"取消失败"。三连成型。
2. **唯一键 uk_student_course**。`bookCourse` 只查 `status IN (booked, confirmed)`，忽略 `cancelled` 行。INSERT 直接撞唯一键。
3. **登录走了 mock fallback**。`authService.login` 数据库分支抛"账号已被禁用"被外层 try/catch 吞掉，回退到 mock 数据 → mockUser.status=1 → 登录通过。**另外** `models/User.js` 之前在 SELECT 里把 status 硬编码成 `1 as status`，根本读不到真实禁用状态。

### 修改清单（4 个文件）
1. `backend/src/models/User.js` —— `findById` / `findByUsername` 把 SELECT 里的 `1 as status` 改成 `status`，真正读 DB 字段
2. `backend/src/services/authService.js` —— login 内的"用户名/密码错/账号禁用"三类业务错误打 `err.isBusiness=true` 标记，外层 catch 见到该标记直接 throw，不走 mock 兜底
3. `backend/src/services/appointmentService.js`（bookCourse）—— 检查重复预约通过后，再查 `student_id+course_id` 是否存在 cancelled 旧行；若有则 UPDATE 复用 booking_id（status='booked', created_at=NOW()），否则 INSERT。彻底兼容唯一键
4. `frontend/src/pages/MyAppointments/index.vue`：
   - 新增 `cancellingIds: Set<id>` 并用 `:disabled` + `:loading` 防重复点击
   - `handleCancel` 移除 catch 里的 `message.error`，避免与 request 拦截器重复弹

### 验证（curl）
- 禁用 student1 → 登录返回 `respCode 99999, respMsg 账号已被禁用`
- 启用 student1 → 登录 `00000`
- 错密码 → `99999 用户名或密码错误`
- 取消 course 4 后再预约 → `00000 预约成功`（之前 Duplicate）
- 重复预约 → `99998 您已预约过该课程`
- 取消已 cancelled 的预约 → `99998 该预约状态不允许取消`（单条响应）

### 经验沉淀
- 这个项目所有 controller 都包了 try/catch fallback 到 mock，业务错误（"已禁用/无权限/已预约"等）必须打标记跳过 mock，否则一切 fail-safe 都会变成 fail-success。后续动 authService / appointmentService 时要注意这个反模式。
- 前端 axios 拦截器已经统一弹 message.error，业务侧 catch 不要再弹一次，否则会出现叠加。除非业务上明确想 override 拦截器的提示。
